1. TL;DR

xvision and the two reference projects are not in the same product category. Freqtrade is a mature retail crypto bot. Lumibot is a Python framework for retail equities/options/crypto with hosted-deployment add-ons. xvision is a Rust agentic-trading research stack whose differentiator is the four-stage pipeline (Intern → Trader → Risk → Executor), the Karpathy-style autoresearcher, and on-chain identity/reputation via ERC-8004 on Mantle.

The honest comparison: xvision is statistically more rigorous than either reference (paired bootstrap CIs on Δ-Sharpe, deterministic risk gate as a first-class stage, content-addressed lineage with pre-committed evaluation gates) but operationally far thinner. Strategy authoring is a single-method Algorithm trait with no lifecycle hooks. There's no Telegram/Discord, no FreqUI-equivalent dashboard, no hyperopt, no walk-forward harness, no protections framework, no per-trade state mechanism, no informative-pairs / multi-timeframe pattern, no TradingFee abstraction with maker/taker/per-contract knobs.

The most useful things to mine — in priority order — are: (1) Freqtrade's hyperopt architecture (NSGA-III + IHyperOptLoss trait) as a reference for the autoresearcher's parameter-only mutation sub-mode; (2) Freqtrade's protections framework as a model for extending the deterministic risk layer; (3) Lumibot's lifecycle methods + Freqtrade's callback surface as templates for a richer Algorithm trait; (4) both projects' fee/slippage modeling; (5) Freqtrade's config-reload + REST control plane patterns for the engine API. Things to skip: FreqUI (xvision is building its own), Telegram (wrong audience), FreqAI (the LLM Trader is the ML signal), Lumibot's broker grab-bag (Alpaca + Orderly is intentional).

Cycle model

The unit of work is a cycle (renamed from setup on 2026-05-10 — see CLAUDE.md §Terminology and migration 0002_rename_setup_to_cycle.sql). One cycle_id (ULID) ties together a briefings row, a decisions row (one per arm), a risk_outcomes row, an executions row, and any number of traces rows. SQLite via sqlx with raw query strings (no compile-time-checked queries yet). Two migrations live: 0001_init.sql, 0002_rename_setup_to_cycle.sql.

Eval surface

xvn ab-compare --cycles N loads N pinned scenarios. For each: Stage 1 produces one InternBriefing; every Algorithm impl reads that briefing (TraderArm) or computes independently (classical baselines); decisions are passed through Risk; Execution simulates fills. Metrics — Sharpe (primary), max drawdown, profit factor, win rate — computed with paired-bootstrap 95% CIs (block size 8, time-series-correct). The trait surface is minimal:

#[async_trait]
pub trait Algorithm: Send + Sync {
    fn name(&self) -> &'static str;
    async fn decide(&self, snapshot: &MarketSnapshot) -> Option<TraderDecision>;
}

Identity and on-chain (ERC-8004)

Three planned registries (per ADR 0008): IdentityRegistry (per-strategy NFT mint, agentURI → token_id), ReputationRegistry (per-cycle performance feedback with hash anchoring), ValidationRegistry (signed-oracle receipts for autoresearch mutations). Mantle Sepolia testnet + mainnet contract addresses pending SLF2. IdentityClient is a working alloy-v2 client wired against the ERC-8004 draft headers; integration tests are #[ignore] and run only against local anvil. No mint-identity binary ships — operators currently have to write a thin driver against crates/xvision-identity/src/client.rs (see MANUAL.md M7).

Autoresearcher (AR-1, AR-2, AR-3)

Spec at docs/superpowers/specs/2026-05-09-karpathy-autoresearcher-design.md (locked 2026-05-09). Three plan files split by scope:

  • AR-1 — 2026-05-09-autoresearcher-1-mutator-lineage-gate-seal.md. New module xvision-engine/src/autoresearch/. Implements: Haiku-driven mutator that proposes prose/param/tool diffs against a parent StrategyBundle, paper-tests on day + held-out windows, runs deterministic Δ-Sharpe gate with pre-committed ε, commits to a content-addressed lineage at ~/.xvn/lineage/blobs/<blake3>.json, emits a CycleSeal with Merkle root + operator Ed25519 signature. Hard upstream: the eval-engine plan 2026-05-08-eval-engine-plan.md. Wk-2 milestone.
  • AR-2 — cycle orchestrator + LLM judge (Sonnet, metrics-blinded) + canary (sabotaged-parent nightly) + inversion-pair eval + embedding-divergence diversity-decay metric. SSE event emission lands here.
  • AR-3 — Next.js dashboard surfaces: live evening-cycle viewer, genealogy tree (D3 force-directed), mutation diff inspector, mutator-skill ladder.

Status: spec locked, plans drafted, none merged. The hackathon's load-bearing feature is the AR-1 → AR-2 path; AR-3 is the visible UI.

What was lost when CV dropped (ADR 0011, 2026-05-07)

The control-vector substrate (steering tensors, learned vector configurations, vector-trained Qwen3 trader, vectors_off/vectors_on agent manifests) moved to a sibling repo xvision-play. xvision's edge thesis is now: balanced Intern briefing + LLM judgment + evolutionary selection via the autoresearcher + on-chain reputation as a verifiable track record. Whether (a) Intern bias-control + (b) autoresearch selection + (c) on-chain receipts is a sufficient differentiator is the open question — and the hackathon's job is to demo the receipts story strongly enough that the underlying performance claim becomes secondary to the verifiability claim.

Operator UX, briefly

Config: TOML in config/ (default.toml, risk.toml, whitelist.toml). Env override syntax: XVN_<SECTION>_<KEY>. Secrets via env vars or 1Password (op read 'op://Personal/...'). Modes: backtest | paper (Alpaca) | live (Orderly). Logging: tracing with env-filter; traces persisted to SQLite. Dashboard: Axum + Next.js (frontend foundation in worktree, frontend/web/). Kill switch (xvn kill --all < 1s) is on the hackathon roadmap. Docker: ghcr.io/latentwill/xvision with :latest and :identity tags.

3. Freqtrade inventory

Freqtrade is a Python crypto bot, ~25k stars, mature, opinionated. Strategy is a single IStrategy subclass. Backtest, hyperopt, dry-run, and live all run through the same freqtrade CLI. The control plane (Telegram + REST + FreqUI) is unusually complete for an OSS algotrading project.

Backtest

  • Window: single contiguous --timerange per run. No native walk-forward or cross-validation — you script multiple runs.
  • Intra-candle realism: --timeframe-detail 5m on a 1h strategy fires custom_exit/custom_stoploss callbacks every detail candle (~12× the parent in this case). Useful for honest stop-loss simulation.
  • Slippage: none modeled in backtest — orders fill at requested price as long as it's inside the candle's high/low. Live uses a 5% market-order slippage cap. Documented as a known limitation.
  • Fees: single --fee FLOAT ratio, applied entry + exit. Default = ccxt-reported exchange fee.
  • Multi-strategy compare: --strategy-list S1 S2 S3 prints a STRATEGY SUMMARY table side-by-side. No paired statistical comparison.
  • Reporting: SUMMARY METRICS includes CAGR%, Sharpe, Sortino, Calmar, SQN, profit factor, expectancy + expectancy ratio, max % underwater, max drawdown, drawdown duration, rejected-entry signals, max consecutive wins/losses, long/short split. Add --breakdown {day,week,month,year,weekday} for sub-period rollups.
  • Artifacts: zip with backtest report JSON, market change .feather, copy of strategy + params + sanitized config. Caching --cache {none,day,week,month}.
  • Hyperopt: Optuna's NSGAIIISampler. Search-space DSL via class attributes — IntParameter(low, high, default, space=), DecimalParameter(low, high, decimals=, …), RealParameter, CategoricalParameter, BooleanParameter. space= is one of buy, sell, enter, exit, roi, stoploss, trailing, protection, trades, or custom.
  • Hyperopt loss functions (built-in subclasses of IHyperOptLoss): ShortTradeDurHyperOptLoss (default), OnlyProfitHyperOptLoss, SharpeHyperOptLoss, SharpeHyperOptLossDaily, SortinoHyperOptLoss, SortinoHyperOptLossDaily, CalmarHyperOptLoss, MaxDrawDownHyperOptLoss, MaxDrawDownRelativeHyperOptLoss, MaxDrawDownPerPairHyperOptLoss, ProfitDrawDownHyperOptLoss, MultiMetricHyperOptLoss. Custom losses subclass IHyperOptLoss.
  • Edge module: removed (no longer in current docs).

Strategy authoring

Three vectorized methods (called once per backtest pass, each receives dataframe, metadata):

populate_indicators(self, dataframe, metadata) -> DataFrame
populate_entry_trend(self, dataframe, metadata) -> DataFrame  # adds enter_long / enter_short cols
populate_exit_trend(self, dataframe, metadata) -> DataFrame   # adds exit_long / exit_short cols

Plus multi-timeframe via the @informative('1h') decorator or informative_pairs() + merge_informative_pair() (lookahead-safe). Plus per-trade state via trade.set_custom_data(key, value) / trade.get_custom_data(key), JSON-serialized into the Trade row.

Protections framework — composable, declarative:

  • StoplossGuard — pause trading if N stop-loss hits in a window.
  • MaxDrawdown — pause if portfolio drawdown exceeds threshold.
  • LowProfitPairs — blacklist pairs whose recent expectancy is below threshold.
  • CooldownPeriod — N-candle cooldown after exit.

Pairlist pipeline — composable handlers + filters in config: StaticPairList, VolumePairList, PercentChangePairList, MarketCapPairList; filters AgeFilter, OffsetFilter, PerformanceFilter, PriceFilter, SpreadFilter, RangeStabilityFilter, VolatilityFilter, FullTradesFilter.

FreqAI — strategy adds feature_engineering_expand_all/basic/standard + set_freqai_targets; framework handles training, walk-forward retraining, prediction. Auto-expands features over indicator_periods_candles × include_timeframes × include_corr_pairlist × include_shifted_candles. Models: LightGBM, XGBoost, CatBoost (regression / classification / multi-target); PyTorch + RL under separate Docker tags.

Operator UX

  • Config: JSON with comments + trailing commas tolerated. Multiple --config files merge in order. Env override prefix FREQTRADE__ with __ as nesting separator (e.g. FREQTRADE__EXCHANGE__KEY, FREQTRADE__TELEGRAM__CHAT_ID).
  • Telegram: ~25 commands. Most useful: /start /stop /pause /status /profit /balance /forceexit /forceenter /reload_config /performance /trades /count /locks /daily /weekly /whitelist /blacklist /show_config /logs /version /marketdir /stats.
  • REST API: JWT auth. /ping /start /stop /pause /reload_config /status /count /trades /trade/<id> /forceenter /forceexit /profit /performance /balance /daily /weekly /monthly /stats /locks /whitelist /blacklist /pair_candles /pair_history /plot_config /strategies /version /sysinfo /health /logs. WebSocket at /api/v1/message/ws streams entry/exit fills + indicator data.
  • FreqUI: Login (JWT), live trade list with start/stop/force-entry/force-exit, multi-bot dashboard, wallet balance chart, per-pair candle plots driven by plot_config, in-app Plot Configurator. Webserver mode adds data download, pairlist tester, backtest runner with result comparison, historic backtest viewer.
  • Hot-reload: /reload_config re-reads JSON config and restarts the strategy in-process. Strategy code changes still require restart.
  • Logging / health: --logfile rolling 10×1MB, internals.heartbeat_interval (60s), internals.sd_notify for systemd, GET /health with last bot-loop timestamp, GET /sysinfo, GET /logs.
  • Deploy: Docker is the sanctioned path. ghcr.io/freqtrade/freqtrade:stable + :stable_freqai + :stable_freqaitorch + :stable_freqairl. Provided compose files including a freqai variant with optional GPU section.
  • Quickstart: freqtrade new-strategy --strategy MyStrat --template {minimal,full,advanced}, freqtrade new-config, freqtrade install-ui, freqtrade download-data.

4. Lumibot inventory

Lumibot is a Python framework, ~1.4k stars, GPL-3.0, commercially backed by Lumiwealth. Strategy class runs in both backtest and live without code changes. Center of gravity: equities + options + ETFs through retail US brokers. Crypto and futures exist but are thinner.

Backtest

Entry point is a class method on the strategy:

from lumibot.backtesting import YahooDataBacktesting
from datetime import datetime
from lumibot.entities import TradingFee

result = MyStrategy.backtest(
    datasource_class=YahooDataBacktesting,
    backtesting_start=datetime(2023, 1, 1),
    backtesting_end=datetime(2024, 1, 1),
    benchmark_asset="SPY",
    parameters={"symbol": "AAPL"},
    buy_trading_fees=[TradingFee(percent_fee=0.001)],
    sell_trading_fees=[TradingFee(percent_fee=0.001)],
)
  • Data sources: YahooDataBacktesting, PolygonDataBacktesting, ThetaDataBacktesting (options + stocks, recommended), AlpacaBacktesting, TradierData, PandasData (BYO CSV/DataFrame), InteractiveBrokersBacktesting. Selection also via BACKTESTING_DATA_SOURCE env var with a JSON map for per-asset-class routing.
  • Fees: TradingFee(percent_fee=, flat_fee=, per_contract_fee=, maker=, taker=) — clean abstraction. Attached as buy_trading_fees=[…] / sell_trading_fees=[…].
  • Slippage: implicit — fills at next-bar open or limit/stop. No slippage_bps knob.
  • Walk-forward / sweeps: none built in (a community fork exists specifically because of this gap).
  • Multi-asset: stocks, options (full chains, greeks, expirations via get_chains / get_strikes / get_greeks), futures (continuous-contract handling), crypto, FOREX. One backtest can mix asset types.
  • Result artifacts: quantstats_lumi HTML tearsheet (Sharpe, total return, drawdown, monthly heatmap), interactive equity-curve HTML, trades CSV, stats CSV, settings dump, returns DataFrame.

Strategy authoring

Subclass lumibot.strategies.Strategy. Lifecycle methods, all overridable:

initialize(**parameters)             # once at startup
before_starting_trading()            # once per trading day, after market data available
before_market_opens()                # N min before open (default 60)
on_trading_iteration()               # main hook, fires every self.sleeptime
before_market_closes()               # N min before close
after_market_closes()                # post-bell

on_filled_order(position, order, price, quantity, multiplier)
on_new_order(order)
on_canceled_order(order)
on_partially_filled_order(...)

on_abrupt_closing()                  # graceful shutdown
on_bot_crash(error)
on_strategy_end()
trace_stats(context, snapshot_before)  # per-iteration custom metrics → tearsheet

Order placement: two-step order = self.create_order(asset, qty, side, **kwargs); self.submit_order(order). Convenience: submit_orders, cancel_order, cancel_open_orders, sell_all, get_orders, get_order(id). Order types: market, limit, stop, stop_limit, trailing_stop. Order classes: simple, bracket, oco, oto, multileg.

Portfolio: self.cash, self.portfolio_value, self.positions (dict), get_position(asset), get_historical_account_value(), per-asset get_last_price, get_quote, get_historical_prices. Persistent state: self.vars.foo (pickled to disk in live mode).

Parameters: class-level parameters = {…} dict. Override at backtest/live time via parameters={…} kwarg. Reads via self.parameters["key"]. Live mutation via self.set_parameters({…}). No schema/range/distribution declaration — purely freeform — which is exactly why Lumibot has no native hyperopt.

Operator UX

  • Brokers: Alpaca, InteractiveBrokers (REST), InteractiveBrokersLegacy (TWS), Tradier, Schwab, Tradovate (futures), Bitunix (perp futures), Ccxt (Coinbase, Binance, Kraken, Kucoin), and CCXT-shortcut Kraken/Coinbase.
  • Configuration / secrets: .env auto-loaded, plain env vars, or Python config dict. No YAML schema. Per-broker env vars documented in deployment docs.
  • Dashboards / monitoring: none in OSS. Discord webhook via self.log_message(…, broadcast=True). No Telegram, no FreqUI-equivalent. The hosted equivalent is BotSpot (paid).
  • Logging: stdlib logging routed through self.log_message(message, color=, broadcast=). Trader(logfile="logs/test.log").
  • Deploy: Render (~$7/mo, one-click "Deploy to Render"), Replit (~$25/mo), Docker (no first-party compose), BotSpot (Lumiwealth-hosted), local python my_strategy.py.
  • Live/paper switching: flip a broker config flag (ALPACA_IS_PAPER=true, TRADIER_PAPER=True, IB port 7497 vs 7496). No code changes inside the strategy.

6. Recommendations — steal / adapt / skip

steal port directly, low effort, high value. adapt the pattern is right; the implementation needs xvision-specific shape. skip wrong fit, wrong audience, or already covered.

6.1 Backtest / eval

steal The IHyperOptLoss trait + 12 built-in losses. Mirror as a Rust trait HyperOptLoss in xvision-eval: fn evaluate(&self, results: &BacktestResult) -> f64. Implement Sharpe, SharpeDaily, Sortino, SortinoDaily, Calmar, MaxDrawdown, ProfitDrawdown as starter set. The autoresearcher mutator becomes pluggable on which loss it gates on, and operators can pre-commit to one in autoresearch.toml. Critically, this lets the autoresearcher run a parameter-only sub-mode that's recognizable to anyone coming from Freqtrade.

steal --breakdown sub-period rollups. Trivial to add to ab-compare. Per-day / per-week Sharpe + drawdown columns surface regime sensitivity without changing the underlying eval.

steal TradingFee entity (Lumibot). Adopt percent_fee + flat_fee + per_contract_fee + maker + taker as a FeeModel struct. Orderly maker/taker, Alpaca commission-free with regulatory fees — these don't fit a single ratio.

steal Result-bundle zip pattern (Freqtrade). Each ab-compare run produces a zip containing report JSON, the bundle, the sanitized config, and the lineage seal. Already content-addressable; package it for share-with-judges.

adapt Walk-forward harness. Neither reference has one — this is open territory. A minimal version: xvn ab-compare --walk-forward train=N test=M step=K that loops ab-compare over rolling windows and aggregates with the existing paired bootstrap. The autoresearcher's "day window + held-out window" model already implies this; promote it to the eval CLI.

adapt Tearsheet. Generate a quantstats-style HTML at the end of ab-compare. Two paths: (a) emit a JSON summary and post-process via a thin Python wrapper around quantstats_lumi; (b) Rust-native using plotly-rs or static SVGs. Path (a) ships in a day; path (b) is the right long-term answer.

adapt Slippage model. Not modeled by either reference — but xvision's eval-rigor framing implies it should be. Add a SlippageModel trait with implementations None, FixedBps, VolumeProportional, BookImpact (later). Default FixedBps(5).

skip FreqAI. The LLM Trader fills the ML role. Adopting a tabular-ML path adds maintenance with no story benefit.

skip Edge module. Already removed by Freqtrade for being more confusing than useful.

6.2 Strategy authoring

adapt Lifecycle hooks on Algorithm. Don't go full Freqtrade (15+ callbacks is a complexity tax most strategies never benefit from). Add five:

fn on_start(&mut self) {}                            // run once at backtest/live start
fn on_fill(&mut self, fill: &Fill) {}                // post-execution observation
fn on_cycle_end(&mut self, cycle: &CycleSummary) {}  // hook for state updates
fn custom_stoploss(...) -> Option<Bps> { None }       // tighten only
fn confirm_decision(...) -> bool { true }             // last-mile veto inside the algorithm

steal Per-cycle custom data (Freqtrade trade.set_custom_data, Lumibot self.vars). Add a strategy_state table keyed on (agent_id, cycle_id) with a JSON blob. Algorithms get ctx.set("ema_long_at_entry", 4500.0) / ctx.get("ema_long_at_entry"). Cheap, immediately useful for stateful strategies.

adapt Multi-timeframe via @informative pattern. Replicate as a builder on MarketSnapshot: snapshot.with_timeframe(Tf::H1).indicator(Rsi(14)) auto-merges into the active panel with column rename, lookahead-safe by construction. Useful for the autoresearcher to mutate which timeframes a strategy looks at.

adapt Protections framework as a structural extension of Risk. xvision-risk already has the right shape. Add four primitives modeled on Freqtrade: StoplossGuard, MaxDrawdown, LowProfitArm, CooldownPeriod. These compose under the existing deterministic-rule model and slot into config/risk.toml with no architectural change.

adapt Pairlist filters for Mantle perp markets. Most of Freqtrade's pipeline doesn't apply (no orderbook spread for Orderly until phase 11.5), but VolumeFilter + VolatilityFilter + AgeFilter are straightforward and relevant to autoresearch's universe selection.

steal xvn new-strategy --template {minimal,full,onchain,llm} scaffolder. Generates strategies/<name>.md + a stub Algorithm impl + a config snippet. Enormous adoption-friction reducer.

skip Position-adjustment / DCA hooks. Fits awkwardly with the cycle model (one decision per cycle by design). Defer until there's a concrete strategy that needs it.

6.3 Operator UX

steal Hot config reload. xvn reload-config over the engine API. The trader-stage prompt is in config; iterating a prompt without restarting the agent loop is a meaningful operator-UX win.

steal Health endpoint. GET /health with last cycle timestamp, last fill timestamp, last autoresearch session id, current arm count. Mirrors Freqtrade exactly; fits engine-api-foundation.

steal Nested env-override syntax. Move from XVN_SECTION_KEY to XVN__SECTION__KEY (double-underscore separator) so that nested config keys round-trip cleanly. Trivial change in the config loader.

adapt REST control surface. The engine-api-foundation plan covers this, but the Freqtrade endpoint set is a good shopping list: /start /stop /pause /reload-config /status /trades /forceexit /performance /balance /sysinfo /health /logs map cleanly. WebSocket / SSE for fills + autoresearch events.

adapt Trace stats hook (Lumibot's trace_stats). Inside an algorithm, self.trace("custom_metric", value) writes into the traces table and surfaces in the tearsheet. Bridges arbitrary-strategy-metric to dashboard for free.

adapt Dashboard plot_config pattern. Strategies declare which indicators to overlay on the cycle viewer. Mirrors Freqtrade's plot_config; lets a marketplace strategy author control how their work is visualized in AR-3.

skip Telegram bot. Wrong audience for a marketplace play; better signal-to-noise from a Discord webhook or a real dashboard.

skip FreqUI fork. The Next.js frontend is the right answer; FreqUI's UX vocabulary (single-bot retail crypto trader) doesn't transfer.

skip Lumibot's broker grab-bag. Alpaca + Orderly is intentional. IB / Schwab / Tradovate / Bitunix add maintenance with no marketplace-story benefit.

6.4 Cross-cutting

adapt JSON-with-comments configs (Freqtrade). TOML is fine, but the merge-multiple-configs pattern (--config base.toml --config overrides.toml --config secrets.toml) is operationally pleasant and not currently shipped. Layered loading also makes an autoresearch.toml + session-commitment.toml split feel natural.

adapt Pre-commitment file (specifically the autoresearcher's session commitment) as Freqtrade-style human-readable. JSON with comments + trailing commas tolerated, so judges can read the operator's signed pre-registered hypotheses without a viewer.

7. The xvision differentiators worth foregrounding

Things neither reference has, ranked by how load-bearing they are for the marketplace story:

  1. Pre-committed evaluation gates. Operator signs the held-out window + ε threshold + loss-function choice before the autoresearch session runs. Fundamentally changes the trustworthiness of any reported result. Neither FT nor LB has anything like this.
  2. Content-addressed lineage with Merkle-rooted CycleSeal. Each surviving mutation is anchored. Combined with on-chain reputation receipts, the result is a strategy whose history is verifiable, not just claimed.
  3. Non-custodial trading via scoped Ed25519 key. xvision never holds capital. This is the ethical/structural counter-argument to FTX, and it makes the marketplace story coherent: you're delegating trading authority, not custody.
  4. Risk gate as a first-class pipeline stage with deterministic verdicts. FT and LB push position limits, drawdown circuit breakers, etc. into the strategy code. xvision separates them. This is the right architecture; the protections framework can be ported as primitives inside the existing stage rather than scattered around the strategy.
  5. Intern → Trader split. Neutral evidence prep before judgment is a structural anti-bias mechanism. Worth foregrounding as design intent in any positioning material.
  6. Null-result canary, inversion-pair eval, embedding-divergence diversity-decay. Three structural defenses against autoresearch noise-fitting. The phrase "we measured the noise floor of our own eval" is rare in this space.
