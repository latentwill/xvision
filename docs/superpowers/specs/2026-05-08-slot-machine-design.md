## 1. What it is

The **slot machine** is a meta-strategy that uses LLM-driven *ideonomy operators* to generate trading strategy variants from a base thesis. "Ideonomy" here refers to the operator catalog distributed as the `ideonomy-plain` Claude skill — a set of ideational transformations (expand, vary, invert, recombine, juxtapose, generalize, specialize, etc.) for systematically mutating an idea.

It runs in two modes that share the same slot pipeline:

- **Generator mode (factory).** Each "pull" of the lever produces a fresh strategy bundle: a new strategy file with thesis, entry/exit rules, parameter ranges, and regime fit. Output is N candidate strategies routed into the eval engine. Used during research and Strategy-Loom evening cycles to widen the candidate pool faster than humans can author by hand.
- **Arm mode (tradeable).** A pre-generated pool of variants is loaded; on each rebalance window the slot machine samples one variant and trades it for that window. Functions as a control arm in `xvn ab-compare` — analogous to `random_direction` but with ideonomy-derived rules instead of coin flips.

Both modes are LLM-required and conform to the engine's "all strategies have at least one LLM agent" rule from the engine design.

## 2. Why it is worth evaluating

The slot machine is interesting precisely because it is **not** evaluated like a normal strategy:

- A normal strategy has one equity curve. The slot machine produces a **distribution** of equity curves — one per variant.
- The relevant question is not "did this strategy beat buy-and-hold," but two distinct ones:
  1. **Hit rate** — what fraction of variants beat baselines?
  2. **Diversity score** — how spread out is the return distribution? (variance of per-variant returns)
- A slot machine that emits a tight cluster of near-identical variants is broken. One that emits a wide distribution with a heavy positive tail is the goal — the right tail is where novel alpha lives.

This makes the slot machine a tool for **exploring strategy space**, not a tool for trading any single thesis well. It is the closest thing xianvec has to a strategy-discovery engine.

## 3. Architecture

### 3.1 Crate location

New template under the engine MVP layout:

```
crates/xianvec-engine/src/templates/
├── mod.rs                  # existing — registers templates
├── mean_reversion.rs       # existing (engine MVP v1 template)
└── slot_machine.rs         # NEW — this spec
```

Plus a small data module:

```
crates/xianvec-engine/src/templates/slot_machine/
├── mod.rs                  # template impl
├── operators.rs            # ideonomy operator catalog (static text inlined into prompts)
└── variants.rs             # variant pool persistence (read/write JSON under bundles/<id>/variants/)
```

### 3.2 Bundle shape

A slot-machine bundle is a normal `StrategyBundle` (per engine spec) with three LLM slots:

| Slot | Purpose | Mode used in |
|---|---|---|
| `operator_picker` | Given a base thesis and recent pull history, sample 1–3 ideonomy operators to apply on this pull. Outputs a list of operator names + a one-line rationale. | Generator + arm pool prep |
| `variant_generator` | Given the base thesis and the picked operators, emit a structured variant spec: thesis-mutation, entry rule, exit rule, parameter ranges, expected regime, declared inputs. Output validates against the engine's bundle JSON schema. | Generator + arm pool prep |
| `trader` | (Arm mode only) Given the active variant for this window plus the current bar/indicator panel, emit a buy/sell/flat decision. | Arm |

Slot prompts inline the operator catalog text from `ideonomy-plain` at build time. There is **no runtime skill invocation** — the catalog is a `&'static str` in `operators.rs`, regenerated by a small build helper if the upstream skill changes. This keeps the engine self-contained and reproducible.

### 3.3 Determinism and receipts

Every pull is seeded from `(bundle_ulid, slot_ulid, pull_index)`. The seed feeds:

- The operator sampler's PRNG (which operators get picked when the LLM is given a "pick from this set" instruction).
- The variant ID assigned to the output bundle.
- The receipt logged via `xianvec-identity` (ADR 0008) so a pull sequence can be replayed offline.

LLM responses themselves are non-deterministic, but the pull sequence and receipt chain are. Replay reproduces the same pull-by-pull operator picks, regenerates the variant bundles via the same prompts, and re-evaluates. Differences between replays are attributable to LLM stochasticity alone, which is logged.

### 3.4 Variant pool persistence

Generator-mode output:

```
bundles/<slot_machine_id>/
├── bundle.json              # the slot machine bundle itself
├── variants/
│   ├── 00001.json           # variant bundles (full StrategyBundle each)
│   ├── 00002.json
│   └── ...
└── receipts/
    └── pulls.jsonl          # one line per pull: seed, operators, variant_id, ts
```

Arm mode reads from the same pool. A slot machine in arm mode without a populated pool is invalid (validation error at bundle load time).

## 4. Eval engine integration

`xvn ab-compare` gains a `--slot-machine <id>` flag with the following semantics:

1. Load the slot machine bundle.
2. If pool is empty or `--regenerate` is passed, run generator-mode for K pulls (default `K=20`, configurable in the bundle).
3. Evaluate each variant against the same data window the other arms in the comparison are using.
4. Report:
   - Per-variant equity curve, Sharpe, max drawdown.
   - **Hit rate** vs each named baseline (e.g., "12/20 variants beat `ma_crossover`").
   - **Diversity score** (stdev of per-variant total return).
   - Best variant, worst variant, median variant.
   - Pull-receipt summary so the run is replayable.

The slot machine bundle itself does not produce a single equity curve. Tools that demand one (e.g. naive ranking) get the median variant by default; the bundle declares this behavior in its public manifest.

## 5. Operator catalog scope

The `operators.rs` module ships an inlined snapshot of the `ideonomy-plain` operator catalog. For MVP we restrict to a curated subset that maps cleanly onto trading-strategy mutations:

- **expand** — add a confirming signal (e.g. add a volume filter to a pure-price strategy)
- **vary** — change a parameter range (e.g. RSI(14) → RSI(7..21) sweep)
- **invert** — flip the decision direction (mean-reversion ↔ momentum)
- **recombine** — splice two existing baselines' rules
- **juxtapose** — apply two contradictory signals and trade only when they agree
- **specialize** — restrict to a specific regime (e.g. only when funding is negative)
- **generalize** — drop a constraint (e.g. remove the volume filter)

Operators outside this list (perspective shifts, metaphor, narrative reframing) are excluded in MVP because they don't translate into structured strategy specs without a lot of LLM hand-holding. Future versions may broaden the catalog.

## 6. Out of scope for this spec

- Multi-asset / multi-timeframe variants. MVP slot machine generates single-asset, single-timeframe strategies matching the asset/timeframe of the base thesis.
- Variant pool curation / pruning. The pool is append-only in MVP; no auto-eviction of low performers.
- Online learning / pull-policy adaptation. The operator picker does not learn from past variant performance in MVP. Adding a reward signal is a clear future direction but not v1.
- Marketplace publishing of slot machines as products. Treated as an internal research tool first; marketplace surface deferred.
- Native Rust port of ideonomy operators. Catalog stays as inlined text driving the LLM, per the LLM-required design rule.

## 7. Backlog entry and workflow

Per the strategies/README.md format, the slot machine gets a folder under `strategies/slot_machine/` with:

- `README.md` — index page listing the variant generators worth exploring (different base theses, different operator subsets).
- One `<base_thesis>.md` per scoped slot-machine instance (e.g. `mean_reversion_slot.md`, `funding_skew_slot.md`).

A new index subsection is added to `strategies/README.md` titled **"Queued — Idea generators"** to distinguish meta-strategies from base strategies.

Status starts at `idea`; promotes to `queued` once we have at least one scoped base-thesis file with a parameter-and-operator selection.

## 9. Open questions for review

- Is `K=20` pulls the right default, or should generator mode default to fewer (e.g. 5) to control LLM token spend?
- Should the operator picker be allowed to "pull from the same operator twice in one pull" (e.g. `vary` + `vary` with different parameters)? MVP says no for clarity; flag for future.
- Should arm mode hold the same variant for N windows (variant persistence) rather than resampling every window? MVP samples every window; this may produce excessive turnover.
