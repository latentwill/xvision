# Filter v1 — shape spec

**Date:** 2026-05-21 (revised; supersedes the earlier wider draft of the same
date that included a typed `SlotRuntime` split, `AgentKind`, and a graph
executor — those are explicitly v1.5+, see "Out of scope" below).
**Status:** Ready for conductor pickup. Build-ready v1.

**Intake link:** `team/intake/2026-05-21-eval-honesty-and-agent-graph.md` →
`agent-graph-composition` track. v1 of this spec covers the deterministic
indicator branch only; the LLM-backed filter branch + optimizer hook is
covered by `team/intake/2026-05-21-dspy-dsrs-optimizer-adoption.md` (Stage 1)
and ships as v1.5 on top of v1.

## Goal

Add Filters as a first-class platform primitive: deterministic per-bar
scanners that cheaply gate when an LLM agent actually runs. v1 ships the
deterministic-indicator branch only — no LLM filters, no graph executor, no
`SlotRuntime` split. That keeps v1 small, fast to land, and validates the
operator UX (the Filter entity, the activation modes, the wake-up event
shape) before any LLM-backed work expands the surface.

The headline cost win is real: ~20× token reduction at a 5% gate hit-rate.
See the exploration notes archived on the `glamin-cortex-explorations`
worktree for the math (kept off main per operator direction). The
2026-05-19 spec (`docs/superpowers/specs/2026-05-19-eval-evidence-and-agent-filters.md`)
established the product framing; this spec narrows that proposal to a
build-ready v1.

## Data model

### `Filter`

```rust
pub struct Filter {
    pub id: FilterId,                        // ULID
    pub strategy_id: StrategyId,
    pub display_name: String,
    pub description: Option<String>,
    pub status: FilterStatus,                // Draft | Active | Paused | Archived
    pub asset_scope: Vec<Symbol>,            // v1 ceiling: 1; multi-symbol is v1.5
    pub timeframe: Timeframe,
    pub scan_cadence: ScanCadence,           // v1: BarClose only
    pub conditions: ConditionTree,           // see below
    pub cooldown_bars: u32,                  // 0 = none
    pub max_wakeups_per_day: Option<u32>,    // None = unlimited
    pub wake_when_in_position: WakeInPosition,  // Always | OnInvalidationOrTargetOnly | Never
    pub agent_context_template: AgentContextTemplateId,  // v1 ships one: "compact_trade_context_v1"
}
```

### `ActivationMode` (strategy-level)

```rust
pub enum ActivationMode {
    EveryBar,           // today's behavior; LLM runs each cadence-gated bar
    FilterGated,        // LLM runs only on filter wake-ups (and per WakeInPosition)
    CompiledRules,      // out of v1 scope; documented but not implemented
}
```

A `Strategy` gains `activation_mode: ActivationMode` defaulting to `EveryBar`
for back-compat. `CompiledRules` parses but `Strategy::validate()` returns
`E_FILTER_ACTIVATION_MODE_NOT_IMPL` if selected; the variant is reserved so
the wire format stays stable.

### Condition tree

```rust
pub enum ConditionTree {
    All(Vec<Condition>),       // logical AND across leaf conditions
    Any(Vec<Condition>),       // logical OR
}

pub struct Condition {
    pub lhs: Operand,
    pub op: Operator,
    pub rhs: Operand,
}

pub enum Operand {
    Indicator(IndicatorRef),   // e.g. ema_20, rsi_14, close
    Numeric(f64),
    Range(f64, f64),           // ascending; only valid with Operator::Between
}
```

### Indicator catalog (v1: 6 indicators)

| Id | Math | Required params |
|---|---|---|
| `ema_<n>` | exponential moving average of close, period n | n in [2, 500] |
| `sma_<n>` | simple moving average of close, period n | n in [2, 500] |
| `rsi_<n>` | Wilder RSI of close, period n | n in [2, 100] |
| `atr_<n>` | Wilder ATR, period n | n in [2, 100] |
| `atr_pct_<n>` | ATR as percent of close, period n | n in [2, 100] |
| `close` | close price of the evaluated bar | (none) |

`vwap`, bollinger bands, and MACD are v1.5+. The catalog is closed in v1 — no
plugin surface for new indicators until v1.5 introduces the SlotRuntime split.

### Operator catalog (v1: 8 ops)

| Id | Semantics |
|---|---|
| `>` | strictly greater than (numeric, current-bar) |
| `<` | strictly less than |
| `>=` | greater than or equal |
| `<=` | less than or equal |
| `==` | equality within `f64::EPSILON * 1e6` tolerance |
| `crosses_above` | lhs(t) > rhs(t) AND lhs(t-1) <= rhs(t-1) |
| `crosses_below` | lhs(t) < rhs(t) AND lhs(t-1) >= rhs(t-1) |
| `between` | rhs is `Range(lo, hi)`; lhs in `[lo, hi]` |

## DSL (TOML and JSON)

TOML form, authored or rendered as a "condition card" in the UI:

```toml
[filter]
id = "f_01JX..."
strategy_id = "s_01JX..."
display_name = "Trend Pullback Wake-up"
asset_scope = ["BTC/USD"]
timeframe = "1h"
scan_cadence = "bar_close"
cooldown_bars = 3
max_wakeups_per_day = 2
wake_when_in_position = "on_invalidation_or_target_only"
agent_context_template = "compact_trade_context_v1"

[[filter.conditions.all]]
lhs = "ema_20"
op  = ">"
rhs = "ema_50"

[[filter.conditions.all]]
lhs = "close"
op  = "crosses_above"
rhs = "ema_20"

[[filter.conditions.all]]
lhs = "rsi_14"
op  = "between"
rhs = [50.0, 70.0]

[[filter.conditions.all]]
lhs = "atr_pct_14"
op  = ">"
rhs = 0.6
```

JSON form is the same shape, used over the dashboard API. ts-rs exports keep
the two surfaces in sync.

## Validation rules (10)

Each rule has a stable error code so the frontend can render targeted help.

| # | Code | Rule |
|---|---|---|
| 1 | `E_FILTER_UNKNOWN_INDICATOR` | Every `IndicatorRef` resolves to a v1 catalog entry with valid `n`. |
| 2 | `E_FILTER_UNKNOWN_OPERATOR` | Every `Operator` is in the v1 catalog. |
| 3 | `E_FILTER_OPERAND_TYPE` | Numeric/Range operands match the operator's type contract (e.g. `between` requires `Range`; `crosses_*` requires `Indicator` on both sides). |
| 4 | `E_FILTER_RANGE_ORDER` | `Range(lo, hi)` has `lo < hi` finite. |
| 5 | `E_FILTER_NUMERIC_BOUNDS` | RSI thresholds in `[0, 100]`; ATR% > 0; etc. (per-indicator bounds table). |
| 6 | `E_FILTER_FUTURE_LEAK` | No condition references a future bar; the parser rejects any `+N` index on `IndicatorRef` (none exist in v1, but the rule is explicit so v1.5 plugins inherit it). |
| 7 | `E_FILTER_COOLDOWN_NEG` | `cooldown_bars >= 0`. |
| 8 | `E_FILTER_WAKEUP_CAP` | `max_wakeups_per_day`, if set, is in `[1, 1440]`. |
| 9 | `E_FILTER_ASSET_SCOPE` | `asset_scope.len() == 1` in v1. |
| 10 | `E_FILTER_EMPTY_TREE` | `ConditionTree::All(v)` and `Any(v)` have `v.len() >= 1`. |

Validation runs in two phases: `parse` (DSL -> struct, syntactic errors) and
`validate` (struct -> typed errors, semantic). Both phases produce error
values containing the field path (`filter.conditions.all[2].rhs`) for UI
rendering.

## Runtime semantics

### Hook point

`crates/xvision-engine/src/eval/executor/backtest.rs:496` — the per-bar loop
`for (i, bar) in bars.iter().enumerate()`. The filter evaluator runs **after**
the cadence gate (`if (bar.timestamp.timestamp() / 60) % cadence_min != 0`)
and **before** the LLM dispatch path. The dispatch is conditional on the
strategy's `activation_mode`:

```text
for each cadence-gated bar:
  if activation_mode == EveryBar:
    dispatch_llm(bar)                                    // today's behavior
  elif activation_mode == FilterGated:
    let event = filter.evaluate(bar, indicators_cache)   // deterministic
    record(FilterEventV1 { triggered, suppressed_reason, ... })
    if event.triggered:
      dispatch_llm(bar, wake_reason)
    # else: no LLM call this bar
```

### Per-bar evaluation

Pure function: `evaluate(&Filter, &BarHistory, &mut FilterRuntimeState) -> FilterEventV1`.

`FilterRuntimeState` holds (a) bars remaining in current cooldown, (b)
wake-ups so far today, (c) the previous bar's indicator snapshot for
`crosses_*` semantics, (d) the position state at evaluation time.

Suppression precedence (first match wins):

1. `wake_when_in_position` says skip when position is non-flat.
2. `max_wakeups_per_day` cap reached for the bar's calendar day in
   `Filter.timeframe`'s timezone (UTC in v1).
3. `cooldown_bars` not yet elapsed since the last triggered bar.
4. Conditions evaluate to false -> not suppressed; the event records
   `triggered=false` with no `suppressed_reason`.

### Determinism guarantees

`evaluate()` is pure, deterministic, and does not touch system clocks,
randomness, or the network. The indicator cache is keyed by `(bar_timestamp,
indicator_id)`. Same inputs -> identical `FilterEventV1`. This is the
property the regression fixtures (stage 5 of the plan) verify.

## Export shape

### `FilterEventV1` (one row per cadence-gated bar)

```rust
pub struct FilterEventV1 {
    pub bar_timestamp: DateTime<Utc>,
    pub filter_id: FilterId,
    pub triggered: bool,
    pub suppressed_reason: Option<SuppressedReason>,  // InPosition | DailyCap | Cooldown
    pub conditions_passed: Vec<ConditionId>,
    pub conditions_failed: Vec<ConditionId>,
    pub indicator_snapshot: BTreeMap<IndicatorRef, f64>,
}
```

Emitted to `events.jsonl` per run alongside V2D `memory_*` events and V2E
trace events. Same channel; no new sink.

### `filter_summary` (one per run)

```rust
pub struct FilterSummary {
    pub filter_id: FilterId,
    pub bars_scanned: u32,
    pub wakeups: u32,
    pub suppressed_in_position: u32,
    pub suppressed_daily_cap: u32,
    pub suppressed_cooldown: u32,
    pub llm_calls_saved: u32,                  // bars_scanned - wakeups (in FilterGated mode)
    pub estimated_tokens_saved: u64,           // llm_calls_saved * avg_briefing_token_cost
}
```

Lands in the eval run summary export next to the existing token-economics
block (per the 2026-05-19 spec's "Token efficiency report v1").

## Acceptance criteria (10)

1. **Parse / validate roundtrip.** TOML and JSON form parse to identical
   `Filter` structs. Round-tripping `Filter -> TOML -> Filter` is identity.
2. **Indicator x operator coverage.** Every legal combination of the 6
   indicators x 8 operators x valid operand type produces a `Filter` that
   passes validation. Invalid combinations produce the matching error code
   from the validation table.
3. **Field-path error rendering.** Each of the 10 validation errors carries
   the JSON-pointer path to the offending node (e.g.
   `/conditions/all/2/rhs`). UI tests assert the path is surfaced to the
   user.
4. **Deterministic per-bar evaluation.** A golden-fixture scenario (200
   warmup bars + 100 decision bars on a fixed seed) produces a byte-identical
   `FilterEventV1` sequence across runs. Snapshot stored at
   `crates/xvision-filters/tests/fixtures/`.
5. **FilterGated activation reduces LLM calls.** Same scenario, same
   strategy, `EveryBar` vs `FilterGated`: `FilterGated` issues exactly the
   number of LLM dispatches equal to the filter's `wakeups` count, and zero
   on the others.
6. **Cooldown suppression.** With `cooldown_bars = 3`, a filter that would
   fire on bars 5, 6, 7, 11 fires only on bars 5 and 11. The intervening
   bars produce `FilterEventV1` rows with
   `suppressed_reason = Some(Cooldown)`.
7. **Daily cap suppression.** With `max_wakeups_per_day = 2`, the third+ valid
   trigger in a calendar day records `suppressed_reason = Some(DailyCap)`.
8. **In-position suppression.** With `wake_when_in_position =
   OnInvalidationOrTargetOnly`, any otherwise-valid trigger while the
   portfolio holds a non-flat position records
   `suppressed_reason = Some(InPosition)` unless the trigger condition
   includes a documented invalidation/target signal (out of v1 scope as a
   computed signal; v1 treats this mode equivalent to `Never` and records
   the conservative suppression).
9. **Frontend types compile.** `ts-rs` exports for `Filter`,
   `ActivationMode`, `FilterEventV1`, `FilterSummary` regenerate cleanly;
   `pnpm --dir frontend/web typecheck` passes.
10. **Read-only panels render.** Eval-review run detail surface gains a
    Filter panel that lists `bars_scanned`, `wakeups`, suppression counts,
    `llm_calls_saved`, `estimated_tokens_saved`. Component test asserts the
    panel renders correctly given a fixture `FilterSummary`.

## Out of scope (v1.5+)

The earlier draft of this spec proposed a typed `SlotRuntime` enum,
`AgentKind`, edge graph with `Expr`, schema validation findings, and the
graph executor replacing the linear pipeline. Those remain valid design but
**ship as v1.5** to keep v1 small and validate the operator UX of Filters
as deterministic gates before any LLM-filter complexity lands.

Explicitly deferred:

- **LLM-backed filters** (a filter whose evaluation IS an LLM call against
  a labeled prompt — e.g. "Is this regime trending or chopping?"). v1.5
  introduces this via `SlotRuntime::Llm` wrapping into a `kind: filter`
  `AgentRef`.
- **DSPy / DSRs optimizer hook.** Per
  `team/intake/2026-05-21-dspy-dsrs-optimizer-adoption.md` Stage 1, LLM
  filters compile through DSRs (BootstrapFewShot / MIPROv2) against a
  labeled holdout. Diff surfaces in eval-review; operator opts in. The
  filter v1 wave is the structural prerequisite — without the Filter
  entity and the FilterEventV1 export, there's nothing to optimize and
  nowhere to surface the diff.
- **`AgentKind` enum and edge graph** (`Expr`, `Edge`, graph executor).
  v1.5 introduces `kind: filter` on `AgentRef` and the graph executor
  replacing the linear pipeline. v1 keeps the existing linear pipeline and
  adds Filters as a **gate** on the strategy's `activation_mode`, not as
  pipeline nodes.
- **Schema validation + `SchemaMismatchSummary`.** Requires the
  `SlotRuntime::Llm` path. v1.5.
- **`vwap`, Bollinger bands, MACD**. v1 catalog is closed at 6 indicators.
- **Multi-symbol asset scope.** v1 ceiling is 1; v1.5 lifts.
- **CompiledRules activation mode.** Reserved; not implemented in v1.
- **Filter marketplace / reusable artifacts.** v2.
- **Auto-amendment of agent `system_prompt` driven by Filter findings.**
  Autoresearcher V3 territory; covered by the DSPy intake's Stage 2.
