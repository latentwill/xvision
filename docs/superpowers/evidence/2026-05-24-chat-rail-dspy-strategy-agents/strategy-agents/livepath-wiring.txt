2. GUARDRAIL CALL SITES (deliverable #2)

   WIRED (clean, reachable):
   a. eval-launch preflight (same fn, alongside the gate):
      - guardrails::check_prompt_present  -> missing_prompt
      - guardrails::check_provider_available -> provider_unavailable
        (available set = provider names from the RUNTIME CONFIG, i.e. the
         enabled-provider set; fail-closed on config-load error — NOT the
         strategy's own referenced providers, which would make the check a
         no-op)
      - guardrails::check_missing_tool    -> missing_tool
        (available tools = builtins ∪ manifest.required_tools; required tools
         per slot from diagnostics::required_tools_for(primary_capability))
      Each fired ShortCircuit is surfaced via short_circuit_validation():
      ApiError::Validation("[<code>] <display> — <remediation>"). Because this
      runs before the eval_runs row/obs emitter exist, the typed failure is
      recorded as the refused launch (no phantom obs run spawned).

   b. agent execute — invalid_output_schema after schema-recovery EXHAUSTED:
      File: crates/xvision-engine/src/eval/executor/backtest.rs
      - New macro `emit_schema_short_circuit!($detail)` next to
        `finish_decision_span_error!`. Builds ShortCircuit::InvalidOutputSchema
        { role, expected: "TraderOutput", detail } and emits its
        to_typed_error() payload onto the obs/event stream via
        obs.emit_engine_event(code, Some(span), payload_json).
      - Fired at all FIVE recovery-exhausted branches in the trader
        parse/repair block: schema-missing-field repair failed / no repair
        ctx, malformed-json repair failed / no repair ctx, and the
        non-recoverable failure class. Previously these only wrote a free-text
        decision-span error; now the typed `invalid_output_schema` event
        (mapped to UnifiedPayload::ErrorInvalidSchema) is recorded — never a
        silent/opaque degrade.
      - New helper `trader_role_label(agent_slots)` attributes the
        short-circuit to the trader role.

   c. apply-optimized-prompt (accept/swap) — stale_optimized_prompt:
      File: crates/xvision-dashboard/src/routes/optimizations.rs (accept fn)
      - Before `slot.system_prompt = selected.instruction.clone()`, calls
        guardrails::check_optimized_prompt_fresh(run.slot_name,
        snapshot.signature_hash, run.signature_hash). On a CONFIRMED mismatch
        refuses the swap with DashboardError::Validation{ field: sc.code()
        (= "stale_optimized_prompt"), msg: "<display> — <remediation>" } — so
        the swap is recorded as a refused write, not a silent stale apply.
      - Engine is dspy-free, so "current" = run.signature_hash (the bound
        signature shape at optimization time). When run.signature_hash is None
        the guard is skipped (fires only on confirmed mismatch, never on
        missing provenance).

   LEFT AS MARKER (could not be cleanly reached — see why):
   - strategy_references_unattached_slot: the call
     guardrails::check_slot_attached IS wired in the preflight loop as a
     defense-in-depth regression guard, BUT its trigger is pre-empted upstream:
     `resolve_agent_slots` (which runs before the gate) already errors with
     ApiError::NotFound / "agent has no executable slots" for a missing agent
     or zero-slot agent, AND always maps the agent's first slot to the ref's
     role (so resolved.role == agent_ref.role by construction). Forcing a
     reachable trigger would require bypassing resolve_agent_slots, which would
     be a fragile hook. The check is kept (cheap, fires if resolution semantics
     ever change) but is documented in-code as pre-empted; no integration test
     claims to trigger it.
   - disabled_tool / empty_demo_set / filter_signal_requested_but_absent /
     dashboard_artifact_without_persisted_row: out of this track's scope (live
     eval/agent dispatch). Not wired here; their detectors exist and are
     unit-tested in guardrails::tests. (disabled_tool belongs to the Phase 2.3
     server-side tool-policy check; empty_demo_set to the optimization launch
     route; filter_signal to agent::pipeline edge resolution;
     dashboard_artifact to dashboard mutation persist — each a different
     track/owner.)
