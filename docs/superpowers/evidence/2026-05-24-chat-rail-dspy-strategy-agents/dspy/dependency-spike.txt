DEPENDENCY TREE (the real caveat)
  *** Workspace does NOT use rig-core. *** The live LLM path is raw reqwest HTTP
  (LlmDispatch) + the Cline sidecar — NOT rig-core. rig-core enters ONLY transitively
  through dspy-rs. This corrects the plan's Phase 3.3 "rig-core completion-model adapter"
  assumption: there is no existing rig-core in the runtime to match.

  dep        dspy-rs wants   workspace pins   verdict
  rig-core   0.22.0          (not used)       new transitive dep; no conflict
  tokio      1.46 (full)     1.x (full)       unifies -> one tokio 1.52.1
  serde      1.0.219         1.x              unifies
  reqwest    0.12 (blocking) 0.13             SemVer-incompatible -> two copies coexist
  arrow      56.x            58.3.0           SemVer-incompatible -> two copies coexist
  parquet    56.x            58.3.0           two copies coexist
  + heavy tail: foyer, hf-hub, kdam, rayon, bon (93 new locked packages)

  Multiple major/minor versions coexisting is allowed by Cargo and is NOT a build
  failure. It only matters if dspy-rs types must cross the API boundary with engine
  types on the other version, or for binary bloat / compile time. Contained while
  dspy-rs lives in an isolated, non-default-member crate.
