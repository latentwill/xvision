WHAT WAS WIRED
--------------
1. wizard_loop.rs — pre-tool checkpoint hook (Phase 2.5).
   At the existing pre-WRITE-tool enforcement site in run_one_turn, after
   the policy gate returns Allow and before run_tool executes, a WRITE-class
   tool takes a CheckpointKind::PreTool snapshot via
   Checkpointer::new(pool, xvn_home).snapshot(session_id, PreTool, req).
   - strategy-write tool → SnapshotRequest{ strategy_id (input id/strategy_id
     or last_draft_id), tool_policy:true, focus:true }.
   - attach_agent additionally captures agent_id.
   - returned id persisted via ChatSessionStore::set_checkpoint_head.
   - CheckpointCreated(CheckpointWrittenEvent) queued as a PolicyEvent.
   - FAIL CLOSED: a snapshot or set_checkpoint_head failure blocks the
     mutating tool and feeds a typed denial tool_result back to the model
     plus an ErrorPersistenceFailed event.
   - read-only tools and work-launchers with no draft target take no
     checkpoint.

2. chat_rail.rs — focus load on session start (Phase 2.4).
   In the SSE chat handler's spawned task, after the projector is seeded and
   before WizardLoop::new, focus::load(xvn_home, scope) runs; on a hit the
   XVN_HOME-relative path is recorded via ChatSessionStore::set_focus_path
   and a FocusLoaded(FocusEvent) is appended to the session_events log and
   published on the per-session bus.

3. wizard_loop.rs — per-turn focus injection (Phase 2.4).
   system_prompt() is now async; each turn it reloads the scope's FocusDoc
   (cheap fs read so an operator edit between turns takes effect) and splices
   the content into a clearly-delimited "## Focus" section, queuing a
   FocusInjected(FocusEvent{content_hash}) PolicyEvent.

Both new event kinds ride the existing PolicyEvent / take_policy_events()
projection channel that the chat_rail route already drains and projects via
WizardEventProjector::project_payload — no new plumbing.
