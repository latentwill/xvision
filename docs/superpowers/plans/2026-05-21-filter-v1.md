# Filter v1 — implementation plan

**Date:** 2026-05-21
**Status:** Companion to `docs/superpowers/specs/2026-05-21-filter-v1.md`.
**Scope:** Deterministic-indicator-only v1. LLM-backed filters + DSPy
optimizer hook ship as v1.5 (see `team/intake/2026-05-21-dspy-dsrs-optimizer-adoption.md`).

**Sequencing dependency (added 2026-05-21):** This plan's Stage 2 wires
filter evaluation into the per-bar backtest loop. The Alpaca-Live intake
(`team/intake/2026-05-21-alpaca-live-eval-and-executor-refactor.md`)
sequences an `executor-refactor` ahead of this work that extracts
`BarSource` / `Clock` / `FillSink` traits, collapses to a single
`Executor`, and deletes `PaperExecutor`. **Stage 2 below lands against
the post-refactor unified `Executor`, not against today's
`BacktestExecutor` directly.** The hook-point reference to "line 496 of
`backtest.rs`" in the original v1 draft is therefore stale; the actual
hook point is the per-bar dispatch site in the unified `Executor`.
Stage 1 (pure crate) is unaffected. Stages 3–5 are unaffected.

This plan decomposes the spec into 5 PR-sized stages. Each stage names its
file targets, tests, and verification commands. Stages 1–3 ship serially
(Stage 2 depends on Stage 1's crate; Stage 3 depends on Stage 2's runtime
events). Stages 4 and 5 can ship in parallel once Stage 3 lands.

## Sequencing

```
[executor-refactor lands first — see Live intake]
   └─ Stage 1 (domain + validation)
          └─ Stage 2 (backtest evaluator, against unified Executor)
                 └─ Stage 3 (export + summary)
                        ├─ Stage 4 (frontend types + read-only panels)
                        └─ Stage 5 (regression fixtures)
```

Suggested wave duration: 1.5–2 weeks if a single agent owns end-to-end;
faster if Stages 4 and 5 split between agents.

## Stage 1 — Domain + validation

**Goal.** Land the Filter data model, DSL parsers, and the 10 validation
rules as a standalone crate with zero engine dependencies. No runtime
behavior yet.

**File targets.**

- `crates/xvision-filters/Cargo.toml` (new crate; workspace member;
  dependencies: `serde`, `serde_json`, `toml`, `thiserror`, `chrono`,
  `ulid`, `ts-rs` behind a feature gate matching workspace convention).
- `crates/xvision-filters/src/lib.rs` — module skeleton + re-exports.
- `crates/xvision-filters/src/types.rs` — `Filter`, `FilterId`,
  `FilterStatus`, `ActivationMode`, `ScanCadence`, `WakeInPosition`,
  `ConditionTree`, `Condition`, `Operand`, `IndicatorRef`, `Operator`,
  `AgentContextTemplateId`. All `#[derive(Serialize, Deserialize, TS)]`
  per workspace ts-rs pattern.
- `crates/xvision-filters/src/parse.rs` — `parse_toml(&str) -> Result<Filter>`,
  `parse_json(&str) -> Result<Filter>`. Wraps the `serde` deserializers
  with `ParseError` carrying field-path context.
- `crates/xvision-filters/src/validate.rs` — `validate(&Filter) -> Result<(), ValidationError>`.
  Implements the 10 rules from the spec. Error type is an enum carrying the
  stable error code (`E_FILTER_*`) and a JSON-pointer field path.
- `crates/xvision-filters/src/errors.rs` — `FilterError` + `ParseError`
  + `ValidationError` with `thiserror::Error` derives. Stable error codes
  exposed as `impl ValidationError { pub fn code(&self) -> &'static str }`.
- `crates/xvision-filters/tests/parse_roundtrip.rs` — golden TOML/JSON
  fixtures + assertion that `parse → serialize → parse` is identity.
- `crates/xvision-filters/tests/validate_codes.rs` — one test per
  validation rule asserting (a) the rule rejects a constructed invalid
  Filter, (b) the returned code matches the spec table, (c) the field
  path is the expected JSON pointer.

**Tests.**

- `cargo test -p xvision-filters` covers parse roundtrip and all 10
  validation rules.
- No new integration tests in `xvision-engine`; Stage 1 does not touch
  the engine.

**Verification.**

```bash
cargo test -p xvision-filters
cargo clippy -p xvision-filters -- -D warnings
cargo fmt -p xvision-filters --check
bash scripts/board-lint.sh
```

**Acceptance.**

- All 8 operators and 6 indicators parse from TOML and JSON.
- Every `E_FILTER_*` code in the spec table has a matching test.
- ts-rs exports compile in the workspace's `ts-rs` step; no schema drift
  flagged by the existing ts-rs check (`pnpm --dir frontend/web typecheck`
  is **not** run at this stage — that's Stage 4).
- Workspace builds cleanly: `cargo build --workspace`.

**Risk notes.**

- Filter TOML lives in a string column at this stage (no engine
  storage yet). Stage 2 introduces the column. Stage 1 is a pure crate
  add — the smallest possible PR shape.

## Stage 2 — Backtest evaluator

**Goal.** Wire Filter evaluation into the per-bar backtest loop **of the
post-refactor unified `Executor`**. Add the `activation_mode` strategy
field. Land the runtime state machine for cooldown, daily cap, and
in-position suppression. No exports yet.

**File targets.**

- `crates/xvision-filters/src/runtime.rs` (new) — `evaluate(&Filter,
  &BarHistory, &mut FilterRuntimeState) -> FilterEvaluation`. Pure,
  deterministic. `FilterEvaluation` is the internal type that becomes
  `FilterEventV1` in Stage 3.
- `crates/xvision-filters/src/indicators.rs` (new) — math for the 6
  v1 indicators (`ema_n`, `sma_n`, `rsi_n`, `atr_n`, `atr_pct_n`,
  `close`). Pure functions over `&[Ohlcv]`. Note: `xvision-mcp/src/tools.rs`
  already has `indicator_panel` math; v1 copies the minimum subset here
  and the dedup-into-`xvision-indicators` extraction is a v1.5 chore.
- `crates/xvision-filters/src/state.rs` (new) — `FilterRuntimeState`
  (per-filter, per-run): cooldown counter, today's wake count + day
  rollover, previous-bar indicator snapshot for `crosses_*`.
- `crates/xvision-engine/src/strategies/model.rs` — add
  `activation_mode: ActivationMode` to `Strategy` with `#[serde(default)]`
  (default = `EveryBar` for back-compat). Add `filters: Vec<Filter>` on
  `Strategy` (default = empty).
- `crates/xvision-engine/migrations/<next>_filter_v1.sql` — add
  `activation_mode TEXT NOT NULL DEFAULT 'every_bar'` and
  `filters_json TEXT NOT NULL DEFAULT '[]'` to the strategies table
  (verify migration number against `team/MANIFEST.md`; contract author confirms).
- `crates/xvision-engine/src/eval/executor/mod.rs` (or the unified
  `Executor` site introduced by `executor-refactor`) — at the per-bar
  dispatch site, branch on `strategy.activation_mode`:
  - `EveryBar` → unchanged.
  - `FilterGated` → for each filter in `strategy.filters`, call
    `evaluate()`; if any triggers (and none of higher precedence suppress),
    dispatch LLM with the `wake_reason`; otherwise skip dispatch.
- `crates/xvision-engine/tests/filter_backtest.rs` (new) — integration
  tests covering (a) `EveryBar` produces identical decisions to the
  pre-filter behavior on a golden scenario, (b) `FilterGated` with a
  filter that fires on 5/100 bars produces exactly 5 LLM dispatches,
  (c) cooldown suppression, (d) daily cap suppression, (e) in-position
  suppression honored.

**Tests.**

- `cargo test -p xvision-filters` — runtime + indicator math unit tests.
- `cargo test -p xvision-engine filter_backtest` — the new integration
  file.
- `cargo test --workspace` — full sweep to catch any regression in
  unrelated paths.

**Verification.**

```bash
cargo test --workspace
cargo clippy --workspace -- -D warnings
cargo fmt --check
bash scripts/board-lint.sh
# Migration smoke: spin up a fresh sqlite, run migrations
RUST_LOG=info cargo run -p xvn -- doctor
```

**Acceptance.**

- Existing strategy fixtures backtest byte-identically (no behavior change
  for `EveryBar` strategies).
- A new test strategy with `activation_mode = FilterGated` and one
  indicator-threshold filter fires on exactly the bars the spec's
  fixture expects.
- Cooldown / daily cap / in-position suppression match acceptance criteria
  #6, #7, #8 from the spec.

**Risk notes.**

- Indicator math is duplicated between `xvision-filters` and
  `xvision-mcp/src/tools.rs`. Acceptable for v1; the v1.5 chore extracts
  `crates/xvision-indicators` (spec §"Out of scope").
- `previous-bar indicator snapshot` for `crosses_*` semantics is the
  trickiest piece. Stage 1's tests should include a `crosses_above` /
  `crosses_below` snapshot fixture so Stage 2 inherits a clear contract.

## Stage 3 — Export + summary

**Goal.** Emit `FilterEventV1` per evaluated bar, aggregate into
`FilterSummary` per run, surface in run exports. No frontend work yet.

**File targets.**

- `crates/xvision-filters/src/events.rs` (new) — `FilterEventV1` +
  `SuppressedReason` + `FilterSummary`. `Serialize` + `TS` derives.
- `crates/xvision-engine/src/eval/events.rs` — extend the `Event` enum
  with `Filter(FilterEventV1)` variant. Reuses the existing
  `events.jsonl` sink (same channel V2D `memory_*` events use).
- `crates/xvision-engine/src/eval/executor/mod.rs` — at the per-bar
  dispatch site, after `evaluate()`, emit the corresponding
  `Event::Filter` row.
- `crates/xvision-engine/src/eval/summary.rs` — extend `RunSummary` with
  `filter_summaries: Vec<FilterSummary>`. Aggregator computes from
  `Event::Filter` rows post-run.
- `crates/xvision-engine/src/eval/export.rs` — include
  `filter_summaries` in the JSON export shape; ensure backwards
  compatibility (field is `#[serde(default)]` so older readers don't
  break).
- `crates/xvision-engine/tests/filter_export.rs` (new) — end-to-end
  test: run a `FilterGated` strategy, assert `events.jsonl` contains
  one `Event::Filter` per cadence-gated bar, assert the export's
  `filter_summaries` block matches expected counts and
  `llm_calls_saved`.

**Tests.**

- `cargo test -p xvision-engine filter_export`.
- Verify `Event::Filter` JSON shape with a serde snapshot test in
  `xvision-filters`.

**Verification.**

```bash
cargo test --workspace
cargo clippy --workspace -- -D warnings
# Inspect a real run's events.jsonl to eyeball the new rows:
cargo run -p xvn -- eval run --strategy <test_strategy_with_filter_gated>
ls -la /tmp/xvn-test-data/agent_runs/blobs/*/events.jsonl
```

**Acceptance.**

- One `Event::Filter` row per cadence-gated bar in `FilterGated` mode.
- `FilterSummary.wakeups + suppressed_in_position + suppressed_daily_cap
  + suppressed_cooldown + (bars_scanned - wakeups - suppressed_*) ==
  bars_scanned`. (i.e., counts reconcile.)
- `llm_calls_saved` matches `bars_scanned - wakeups`.
- `estimated_tokens_saved` uses the v1 constant
  `avg_briefing_token_cost = 50_000` (matches MANUAL.md §scaling
  assumption); the constant lives in `xvision-filters/src/lib.rs`
  marked `pub const AVG_BRIEFING_TOKEN_COST: u64 = 50_000;` with a TODO
  comment for v1.5 to make it per-strategy-measured.

## Stage 4 — Frontend types + read-only panels

**Goal.** Surface `FilterEventV1` and `FilterSummary` in the dashboard
run-detail view. Read-only — no editor in v1.

**File targets.**

- ts-rs regeneration produces:
  - `frontend/web/src/api/types.gen/Filter.ts`
  - `frontend/web/src/api/types.gen/FilterEventV1.ts`
  - `frontend/web/src/api/types.gen/FilterSummary.ts`
  - `frontend/web/src/api/types.gen/ActivationMode.ts`
- `frontend/web/src/features/eval/components/FilterSummaryPanel.tsx`
  (new) — renders the summary row: bars scanned, wake-ups,
  suppression breakdown, tokens saved. No popups (per the
  dashboard's no-popups rule in CLAUDE.md). Renders inline under the
  run detail header.
- `frontend/web/src/features/eval/components/FilterEventTimeline.tsx`
  (new) — per-bar timeline strip: a tick per bar with color coding for
  triggered / suppressed-by-X / not-triggered. Hover shows
  `indicator_snapshot` values. Renders next to the existing decision
  timeline; mobile breakpoint stacks them vertically.
- `frontend/web/src/features/eval/RunDetail.tsx` — mount the two new
  panels conditional on `run.filter_summaries.length > 0`.
- `frontend/web/tests/components/FilterSummaryPanel.test.tsx` (new) —
  Vitest covering: panel renders given a fixture, counts display
  correctly, no-filter case renders nothing.
- `frontend/web/tests/components/FilterEventTimeline.test.tsx` (new) —
  Vitest covering: each suppression reason has a distinct visual marker,
  hover surfaces indicator values, empty events array renders gracefully.

**Tests.**

- `pnpm --dir frontend/web test --run` — Vitest sweep.
- `pnpm --dir frontend/web typecheck` — ts-rs types compile.

**Verification.**

```bash
pnpm --dir frontend/web typecheck
pnpm --dir frontend/web test --run
pnpm --dir frontend/web lint
# Visual check: storybook OR pnpm dev + a known FilterGated run
```

**Acceptance.**

- A run with `activation_mode = FilterGated` shows the panel and
  timeline; a run with `EveryBar` shows neither.
- All three suppression reasons render with distinct markers.
- Component tests cover the fixture shapes exported from
  `xvision-filters`.
- No new popups, modals, sheets, or popovers introduced (per CLAUDE.md).

## Stage 5 — Regression fixtures

**Goal.** Lock the deterministic-evaluation guarantee with golden fixtures
that future PRs would have to consciously update.

**File targets.**

- `crates/xvision-filters/tests/fixtures/scenario_btc_1h_300bars.json`
  (new) — 200 warmup + 100 decision bars on a fixed seed. Stored in
  the repo (small file, deterministic, no external API dep).
- `crates/xvision-filters/tests/fixtures/filter_trend_pullback.toml`
  (new) — the example DSL from the spec.
- `crates/xvision-filters/tests/fixtures/expected_events.jsonl` (new) —
  byte-exact `FilterEventV1` output for the fixture combination.
- `crates/xvision-filters/tests/fixtures/expected_summary.json` (new) —
  byte-exact `FilterSummary` for the same.
- `crates/xvision-filters/tests/golden_determinism.rs` (new) — test
  reads the inputs, runs `evaluate()` for each bar, asserts the output
  matches the expected files byte-for-byte. Failure prints a unified
  diff.

**Tests.**

- `cargo test -p xvision-filters golden_determinism` is the gate.

**Verification.**

```bash
cargo test -p xvision-filters golden_determinism
# Cross-platform check (CI matrix):
cargo test -p xvision-filters golden_determinism --target x86_64-unknown-linux-gnu
cargo test -p xvision-filters golden_determinism --target aarch64-apple-darwin
```

**Acceptance.**

- Golden test passes on macOS and Linux CI.
- Test failure on intentional change shows a readable diff (use
  `pretty_assertions` or `insta` crate, whichever the workspace
  already prefers).
- README in `crates/xvision-filters/tests/fixtures/` documents how to
  regenerate the expected files when a deliberate behavior change
  lands.

## v1.5 follow-up (not in this plan)

After v1 lands, the next wave folds in LLM-backed filters and the
DSPy/DSRs optimizer hook per
`team/intake/2026-05-21-dspy-dsrs-optimizer-adoption.md`. Sketched:

1. **LLM filter runtime.** Introduce `SlotRuntime::Llm` and `kind:
   filter` per the earlier wider spec draft. The graph executor and
   `AgentKind` enum land here.
2. **DSRs workspace dep + spike.** Add `dspy-rs` behind a feature gate;
   one-day spike on `10-gepa-llm-judge.rs` confirms maturity.
3. **Filter optimizer pilot.** One LLM filter (regime detector or
   similar) compiles through DSRs against a labeled holdout; diff
   surfaces in eval-review.
4. **Operator UX.** "Tune this filter" opt-in action on the filter
   detail page; original prompt preserved, optimized variant is a
   named artifact.

The v1.5 wave decomposes after v1 lands. This plan does not
pre-specify its contracts.
