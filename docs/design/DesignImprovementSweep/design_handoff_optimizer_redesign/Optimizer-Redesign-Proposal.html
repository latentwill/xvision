XVN · Design Review·Optimizer surface

Make the Optimizer feel like one machine you can watch and steer.

Today the Optimizer reads as a stack of loosely-related panels with three different ways to start a run, buttons that don't start anything, and tall mismatched columns. This is a proposal to collapse it into a single, legible control surface — one place to launch, one live heartbeat to watch, one history to review.

01

What's working against us today

Seven concrete problems, all traceable to the current layout. The thread running through them: the page never decides whether it's a launcher, a live monitor, or a history book — so it tries to be all three, vertically, at once.

Phantom button

"Start" doesn't start anything

The Configure-run card's primary Start button only runs scrollIntoView() to #optimizer-run-controls further down the page. It looks like the launch action but just scrolls — the actual launch lives in a different component.

Weak affordance

"Run optimizer" doesn't look like a button

The real launch control is a ghost outline (border-gold / text-gold) buried at the bottom of a tall left column. The single most important action on the page reads as a quiet link, in the spot least likely to be seen.

Redundancy

Three places to start a run

Configure-run strip, Schedule strip, and the full Launch form all coexist on one screen — overlapping fields, three mental models for the same task. Operators can't tell which one is authoritative.

Layout

Tall, mismatched columns

The live view is a [300px · 1fr · 260px] grid: a long form on the left, an event log in the middle, a thin "Kept / Next" rail on the right — three different heights that never line up and waste the horizontal axis.

Sprawl

A nine-deep vertical stack

Status, Configure, Schedule, Improvement chart, Flywheel, Live view, Writers, Recent cycles, Recent runs — nine full-width blocks in one scroll. The important live state sits below several config strips.

Duplication

Two "Recent cycles" tables

The standalone RecentCyclesTable panel and the live view's own RecentCyclesSectionFull render the same history twice, plus a separate "Recent runs" list — three near-identical ledgers.

Legibility

The live work is hard to actually see

While a cycle runs, "what is happening right now" is spread across a phase stepper, a text event log, a cost ticker, and a state pill — none of them adjacent, none of them the visual anchor. There's no single image of 14 experiments × 5 regimes lighting up, which is the one thing an operator wants to watch.

02

The redesign: one surface, three honest zones

Collapse nine stacked panels into a single screen with a clear top-to-bottom logic: Command (steer it) → Live (watch it) → History (review it). One primary action. One live heartbeat. One ledger.

1

One command bar with one real primary button

State pill + active-run identity on the left; a single contextual primary on the right — Launch run when idle, Pause / Cancel when running. The button does the thing. No more "Start" that scrolls, no ghost "Run optimizer" hiding in a column.

2

The phase spine becomes the heartbeat

Promote the 7-phase stepper (Briefing → Parent → Writing → Evaluating → Gate → Commit → Finish) to a full-width strip directly under the command bar, with the live elapsed ticker. At a glance you know exactly where the machine is in its loop.

3

A live work band you can actually watch

The experiments × regimes heatmap is the visual anchor — cells light up as backtests finish, animate while testing, flash on failure. The event feed and live-spend ticker sit beside it, equal height, instead of as a third orphan column.

4

Outcomes as a horizontal strip, not a tall rail

Kept / Suspect / Dropped / Top-Δ as four equal KPI tiles across the page. Kills the skinny right column and lines everything up on one baseline.

5

Launch collapses to a single inline drawer

Merge Configure-run + Schedule + Launch-form into one drawer that expands in place from the command bar (no scroll jump). Mode (once / N / until-budget), parent, budget, window, model overrides — all in one model. Collapses to a one-line summary while a run is active.

6

One ledger, deduplicated

Merge the two "Recent cycles" tables and the "Recent runs" list into a single history table with a runs/cycles toggle. One place to look back.

03

What to do, in order

Sequenced so the highest-impact, lowest-risk fixes land first. P0 items are mostly deletion and relocation — they reduce code and confusion at the same time.

P0

Kill the phantom "Start" and surface one real launch button

Remove the scroll-only Start button. Make a single, solid primary Launch run in the command bar that opens the launch drawer (or fires directly with stored defaults). Promote "Run optimizer" out of the ghost column into that one button.

P0

Collapse the three launch surfaces into one drawer

Fold Configure-run, Schedule strip, and the Launch form into a single inline configuration drawer with one mental model. Run-mode (once / N / until budget) and schedule become tabs inside it, not separate page strips.

consolidate~2 dayshigh impact
P0

De-duplicate the history

Remove the second Recent-cycles table and merge Recent-runs into one ledger with a runs ⇄ cycles toggle. Pure deletion — less code, less scroll, zero ambiguity.

delete~½ dayquick win
P1

Rebuild the live zone horizontally

Replace the [300px·1fr·260px] three-column with: full-width phase spine, then a 2-up work band (heatmap + feed/spend) on a shared baseline. Move the "Kept/Next" rail into the horizontal outcome strip.

re-layout~2–3 daysclarity
P1

Make the experiments × regimes heatmap the live anchor

Adopt the heatmap already designed in the Autoresearch mock as the centerpiece of the running state — cells animate while testing, fill on done, flash on fail. This is the single image of "live work being done" the page is currently missing.

When idle, lead with Launch + last-run summary + improvement chart. When running, lead with the live band and demote config to a collapsed line. The page should look obviously different depending on whether work is happening.

04

Beyond the Optimizer — quick wins across XVN

The same instincts that fix the Optimizer pay off elsewhere. A few high-leverage moves the audit surfaced.

Buttons

One button vocabulary

Define a shared primary / secondary / ghost / danger button set and use it everywhere. Today the same action renders as a solid fill in one screen and a ghost outline in another — the Optimizer is the worst case, not the only one.

Layout

Kill the orphan rails

The skinny right-hand column shows up on several surfaces. Prefer horizontal KPI strips on a shared baseline; reserve true sidebars for navigation, not stray stats.

Live state

Every live surface needs a heartbeat

Eval runs, agent runs and the Optimizer all stream events. Give each one a single, prominent "what's happening now" anchor instead of scattering it across a pill, a log and a ticker.

Empty states

Teach on the empty screen

"No runs yet" is a wasted moment. Use first-run states to show the loop and a single obvious call to action, so a new operator understands the product before they have data.

Density

Consistent table rhythm

Tables across Eval, Lists and the Optimizer use different row heights, alignments and number formatting. Lock one tabular scale (the Geist Mono tnum already in the system) and apply it everywhere.

Hierarchy

One headline per screen

Several pages open with three competing bold blocks. Commit to a single page headline + supporting line, then let the work fill the space below it.
